# 「今天也没废」网站应用实现计划

## 摘要

将现有基础脚手架（React + TypeScript + antd-mobile）重写为完整的「今天也没废」低电量恢复社区应用。基于PRD文档实现MVP核心闭环：**选择身份 → 选择状态 → 生成摆烂许可证 → 微任务领取 → 社区互动**。技术核心是用前端Canvas渲染3套风格的情绪卡片（公文风/诊断书风/游戏风），MVP阶段不依赖后端服务，用localStorage模拟数据持久化。

## 当前状态分析

- `src/App.tsx` 是652行单体组件，实现了与PRD无关的任务管理功能，需完全重写
- `src/index.tsx` 入口文件结构正常，需适配路由
- `package.json` 缺少路由依赖（react-router-dom），状态管理方案待定
- `manifest.json` pages 配置过于简单，需适配实际页面结构
- 无组件拆分、无样式文件、无工具函数、无数据层

## 技术选型决策

| 决策项 | 选型 | 理由 |
|:---|:---|:---|
| 路由 | react-router-dom v6 | React生态标准方案 |
| 状态管理 | React Context + useReducer + localStorage | 零额外依赖，MVP复杂度可控，MiniApp包体积敏感 |
| 卡片渲染 | 原生Canvas 2D API | PRD明确指定前端Canvas渲染方案 |
| AI对话 | 模板化回复系统（MVP） | 不接入真实AI API，按状态索引的三段式回复库 |
| 字体 | 思源宋体(@fontsource/noto-serif-sc) + 系统字体 | 公文风专用宋体，其他风格用系统黑体 |
| 新增依赖 | react-router-dom@6 | 仅此一个运行时依赖 |

## Canvas卡片渲染方案

**核心参数**：逻辑尺寸 750×1000px（3:4比例），物理像素按 devicePixelRatio 缩放，CSS显示 375×500px。

**渲染引擎架构**：
- `renderer.ts` 统一调度 → 根据 style 分发到对应模板
- 每套模板是独立函数，接收 ctx + cardData + 尺寸参数
- 基础绘制工具：text.ts(自动换行/对齐)、stamp.ts(印章)、progress-bar.ts(HP条)、table.ts(表格)、border.ts(边框)

**公文风渲染流程**：米白背景 → 深棕细线边框 → 标题(思源宋体Bold) → 编号 → 持证人 → 分隔线 → 原因正文 → 分隔线 → 许可内容 → 微任务(方括号高亮) → 金句 → 红色圆章(#C41E3A微旋转) → 品牌落款

**诊断书风渲染流程**：纯白背景 → 标题(黑体Bold) → 编号 → 数据表格(浅蓝灰线) → 低值警示红标记 → 诊断结论 → 建议治疗(微任务) → 金句 → 蓝色徽章 → 品牌落款

**游戏风渲染流程**：深蓝黑背景 → 职业名称框(霓虹青边框发光) → 4条状态条(渐变色) → DEBUFF标签(柔和粉) → 今日BUFF(霓虹青) → 主线任务 → 金句(微发光) → 品牌落款

**导出**：canvas.toBlob('image/png', 1.0) → URL.createObjectURL → 触发下载

## 品牌设计规范

三套风格配色：
- **公文风**：背景 #F5F0E8、文字 #2C2C2C、边框 #8B7B6B、印章 #C41E3A、字体 思源宋体
- **诊断书风**：背景 #FFFFFF、文字 #2C3E50、表格线 #B0C4DE、警示 #E74C3C、字体 系统黑体

## 假设与决策

1. **MVP无后端**：所有数据用 localStorage 存储，社区数据用预置示例数据，AI回复用模板匹配而非API调用
2. **状态管理选Context**：不引入额外依赖，若后期复杂度上升可迁移 zustand
3. **字体方案**：公文风用 @fontsource/noto-serif-sc 按需加载，其他风格用系统字体
4. **朋友互证简化**：不依赖微信JS-SDK，用复制链接+本地模拟方式实现裂变流程
5. **卡片尺寸统一**：3:4竖版（750×1000逻辑像素），不实现9:16变体（V1.1再做）
